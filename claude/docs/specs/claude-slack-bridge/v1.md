# Claude Code Slack Bridge

**Spec Version:** v1
**Linear Task:** CLA-14 "Research Slack integration with Claude for mobile workflows" (child of CLA-4 "Mobile Integration")
**Status:** Draft
**Last Updated:** 2026-02-26

---

## Problem Statement

The Claude Code multi-agent workflow (spec, design, feature, fix, QA, PR) is only accessible from a terminal running the Claude CLI. This means users cannot kick off, monitor, or interact with workflows from a phone, tablet, or any device without SSH/terminal access. For longer-running pipelines (feature builds, bulk specs), there is no way to receive notifications or provide input without staying in the terminal session.

A Slack-based bridge would let users trigger and interact with the full command set from any device where Slack runs -- particularly mobile -- while the actual work continues to execute in the dev environment.

## Goals

- [ ] Users can trigger any existing Claude Code slash command from a Slack message and receive results in that Slack thread
- [ ] Users can provide follow-up input (answer clarifying questions, approve specs, give feedback) by replying in the Slack thread, just as they would in the CLI
- [ ] Each command runs in a fully isolated session -- no context bleed between threads or between sequential commands in the same thread
- [ ] The bridge runs inside the existing dev container with no additional infrastructure (no public URLs, no cloud hosting)
- [ ] Users can monitor progress from mobile without needing terminal access

## Non-Goals

- We are NOT building a new Slack app for general-purpose Claude chat -- this is strictly a bridge to existing slash commands
- We are NOT adding new commands beyond what already exists in the workflow toolkit
- We are NOT persisting session state across restarts of the bridge (in-memory only for MVP)
- We are NOT supporting multi-user or multi-workspace scenarios in MVP -- this is a single-developer, single-workspace tool
- We are NOT building a web UI or any interface beyond Slack
- We are NOT modifying the existing slash commands or agents -- the bridge consumes them as-is

## User Stories

**US-1:** As a developer on my phone, I want to type `@bot /spec "user authentication"` in Slack so that a PM spec is generated in my project without needing terminal access.

**US-2:** As a developer away from my desk, I want to reply in a Slack thread to answer the PM agent's clarifying questions so that the spec workflow continues without me returning to my computer.

**US-3:** As a developer, I want each Slack thread to represent one isolated command session so that I can run `/spec` in one channel and `/feature` in another without them interfering.

**US-4:** As a developer, I want to see only essential messages in Slack (command started, input needed, command finished) so that my Slack is not flooded with verbose agent output.

**US-5:** As a developer, I want to run `/feature PROJ-123` from Slack and have it execute the full pipeline (dev, QA, fix, commit) so that I can kick off long builds from mobile and check results later.

**US-6:** As a developer, I want to check what sessions are currently active by typing `@bot status` so that I know what is running.

**US-7:** As a developer, I want sequential commands in the same thread (e.g., after a spec is done, running `/design`) to start with a fresh context so that the next stage is not polluted by the previous stage's conversation history.

## Functional Requirements

- FR-1: The bridge must accept commands via Slack mentions in the format `@bot /command "arguments"` where `/command` is any command from the existing workflow toolkit: `/spec`, `/design`, `/feature`, `/fix`, `/dev-task`, `/qa-review`, `/dev-fix`, `/worktrees`, `/pr`.
- FR-2: Each command invocation must create a new Slack thread (or use the thread where it was invoked). All subsequent messages for that session must stay within that thread.
- FR-3: Each Slack thread must map to exactly one isolated session. There must be no shared state or context between sessions in different threads.
- FR-4: When a user replies in a thread that has an active session, that reply must be forwarded as input to the running session.
- FR-5: When a command finishes and a new command is started in the same thread (e.g., continuing from spec to design), the previous session must be fully closed and the new command must start with a fresh, unpolluted context.
- FR-6: The bridge must post a Slack message at exactly three points during command execution: (a) when a command starts, (b) when user input is required, and (c) when a command completes (with a summary of the outcome). No other messages should be posted.
- FR-7: A `status` command must list all currently active sessions with their thread, command, and how long they have been running.
- FR-8: Sessions must automatically expire after 30 minutes of inactivity (no user input and no output).
- FR-9: Multiple sessions must be able to run concurrently across different threads.
- FR-10: The bridge must connect to Slack using Socket Mode (no public URL or ingress required).
- FR-11: The bridge must run inside the project's dev container, executing commands against the project's working directory.
- FR-12: The bridge must respect Slack API rate limits and not flood the workspace with messages.
- FR-13: When a session requires user input, the Slack message must clearly indicate what input is expected (e.g., "The PM agent is asking: [question]").

## Edge Cases

- User sends a command while another command is already running in the same thread -- the bridge must reject the second command with a clear message ("A session is already active in this thread. Please wait for it to finish or start a new thread.").
- User replies in a thread after the session has expired -- the bridge must inform the user that the session has timed out and suggest starting a new command.
- User sends an unrecognized command -- the bridge must respond with a list of supported commands.
- The CLI process crashes or exits unexpectedly mid-session -- the bridge must detect this, notify the user in the thread that the session ended unexpectedly, and clean up the session.
- Slack message exceeds Slack's character limit (4000 chars for regular messages) -- output must be truncated or split appropriately.
- User sends a command with no arguments where arguments are required -- the bridge must respond with usage guidance for that command.

## UX Constraints

- The bridge must not require any configuration in Slack beyond installing the bot and adding it to channels.
- Command syntax in Slack must mirror the existing CLI syntax as closely as possible (e.g., `/spec "user auth"` in CLI becomes `@bot /spec "user auth"` in Slack).
- Thread replies to the bot must behave identically to typing input in the CLI -- no special syntax required.
- The "input needed" message must include the actual question or prompt from the agent, not just a generic "input needed" notification.
- The completion message must include a meaningful summary (e.g., "Spec written to docs/specs/user-auth/v1.md, status: Ready for Architect" or "Feature pipeline complete: 12 files changed, QA passed, committed on feature/CLA-14-user-auth").

## Acceptance Criteria

- [ ] AC-1: User can type `@bot /spec "feature name"` in a Slack channel and receive a thread with the spec workflow running, including prompts for clarifying questions.
- [ ] AC-2: User can reply in the thread to answer agent questions, and the conversation continues as it would in the CLI.
- [ ] AC-3: User can run `@bot /feature PROJ-123` and the full dev/QA/fix pipeline executes, with only start/input-needed/completion messages appearing in the thread.
- [ ] AC-4: Running two commands in two different threads simultaneously both complete successfully without interfering with each other.
- [ ] AC-5: After a `/spec` completes in a thread, the user can type `@bot /design` in that same thread and it starts a fresh design session with no context from the spec conversation.
- [ ] AC-6: A session that receives no interaction for 30 minutes is automatically cleaned up, and a subsequent reply in that thread informs the user.
- [ ] AC-7: `@bot status` returns a list of active sessions (or "No active sessions").
- [ ] AC-8: The bridge runs inside the dev container and requires only Slack credentials to operate -- no public URLs, no cloud deployment.
- [ ] AC-9: Slack messages from the bridge are limited to start, input-needed, and completion events -- no verbose agent logging appears in Slack.
- [ ] AC-10: If the underlying process crashes, the user is notified in the thread that the session ended unexpectedly.

## Open Questions

- Q1: Should the bridge support configuring different working directories per Slack channel (e.g., #project-a maps to /workspace/project-a), or is a single project directory sufficient for MVP?
- Q2: When a command naturally transitions (e.g., spec prompts "Run /design now? y/n"), should the bridge automatically handle this as a new fresh session, or should the user explicitly type `@bot /design` to start the next stage?
- Q3: What is the desired behavior when the dev container restarts? Should the bridge auto-start, or is manual restart acceptable for MVP?
- Q4: Should there be any access control (e.g., only certain Slack users can trigger commands), or is the assumption that the workspace is private/trusted?
