# Claude Code Slack Bridge

**Spec Version:** v3
**Linear Task:** CLA-14 "Research Slack integration with Claude for mobile workflows" (child of CLA-4 "Mobile Integration")
**Status:** Draft
**Last Updated:** 2026-02-26

---

## Changes from v2

1. **Command/agent modifications now in scope (Non-Goal removed):** Removed the non-goal "We are NOT modifying the existing slash commands or agents -- the bridge consumes them as-is." This project now includes modifying existing commands and agents to support the bridge's execution model. Three specific changes are required:

   - **Foreground execution:** Commands must run in the main thread (foreground) instead of spawning subagents, so the CLI process can stream output directly to the bridge.
   - **Early worktree creation:** `/spec` and `/design` must both create worktrees, not just `/feature`. (Note: `/spec` already created worktrees per v2 FR-5; `/design` now also creates a worktree if one does not yet exist.)
   - **Bulk/parallel mode removed:** Since commands run in the foreground one at a time per CLI process, bulk invocation modes (e.g., `/spec 3` or `/spec PROJ-123,PROJ-456`) are removed. Each command handles exactly one task per invocation.

2. **Spec/design review in Slack (new feature):** When the PM agent finishes a spec draft or the Architect finishes a design draft, the bridge posts the full markdown content into the Slack thread so the user can read and review directly in Slack (including on mobile). The user can then approve or give feedback by replying in the thread. Long documents are split across multiple messages to respect Slack's message size limits.

3. **New user stories US-11 through US-14** added for the command modifications and in-Slack review features.

4. **New functional requirements FR-22 through FR-29** covering foreground execution, worktree creation in `/design`, removal of bulk mode, and in-Slack document posting.

5. **New acceptance criteria AC-14 through AC-20** for the new requirements.

6. **Updated edge cases** for split messages, partial document delivery, and `/design` worktree creation.

## Problem Statement

The Claude Code multi-agent workflow (spec, design, feature, fix, QA, PR) is only accessible from a terminal running the Claude CLI. This means users cannot kick off, monitor, or interact with workflows from a phone, tablet, or any device without SSH/terminal access. For longer-running pipelines (feature builds, bulk specs), there is no way to receive notifications or provide input without staying in the terminal session.

A Slack-based bridge would let users trigger and interact with the full command set from any device where Slack runs -- particularly mobile -- while the actual work continues to execute in the dev environment.

Additionally, the current workflow requires manual worktree management and manual transitions between stages. The bridge should automate the worktree lifecycle so that a feature's spec, design, and implementation all live on the same branch from the start, enabling a seamless path from idea to pull request.

Finally, the existing commands and agents were designed for interactive terminal use, spawning subagents and supporting bulk operations. To work well with the bridge, these commands need to be adapted: running in the foreground for direct output streaming, creating worktrees at earlier stages, and dropping bulk modes in favor of one-command-per-invocation simplicity.

## Goals

- [ ] Users can trigger any existing Claude Code slash command from a Slack message and receive results in that Slack thread
- [ ] Users can provide follow-up input (answer clarifying questions, approve specs, give feedback) by replying in the Slack thread, just as they would in the CLI
- [ ] Each stage of a feature (spec, design, feature) gets its own Slack thread for context isolation, while sharing a single worktree/branch for continuity
- [ ] The bridge manages the worktree lifecycle end-to-end: creation at spec time (or design time if no spec was run), reuse through subsequent stages, and push to remote on completion
- [ ] The bridge runs inside the existing dev container with no additional infrastructure (no public URLs, no cloud hosting)
- [ ] Users can monitor progress from mobile without needing terminal access
- [ ] Existing commands and agents are adapted to run in the foreground (no subagent spawning) so the bridge can stream their output directly
- [ ] Bulk/parallel command modes are removed in favor of one task per command invocation
- [ ] Users can read and review full spec and design drafts directly in Slack without opening files

## Non-Goals

- We are NOT building a new Slack app for general-purpose Claude chat -- this is strictly a bridge to existing slash commands
- We are NOT adding new commands beyond what already exists in the workflow toolkit
- We are NOT persisting session state across restarts of the bridge (in-memory only for MVP)
- We are NOT supporting multi-user or multi-workspace scenarios in MVP -- this is a single-developer, single-workspace tool
- We are NOT building a web UI or any interface beyond Slack
- We are NOT building orchestrated startup (bridge + dev container + tmux) in this iteration
- We are NOT supporting per-channel project directory mapping -- one parent project directory for the bridge
- We are NOT implementing access control or user permissions

## User Stories

**US-1:** As a developer on my phone, I want to type `@bot /spec "user authentication"` in Slack so that a worktree and branch are created for that feature and a PM spec is generated there, without needing terminal access.

**US-2:** As a developer away from my desk, I want to reply in a Slack thread to answer the PM agent's clarifying questions so that the spec workflow continues without me returning to my computer.

**US-3:** As a developer, I want each stage (spec, design, feature) to get its own Slack thread so that conversations stay focused and do not bleed between stages.

**US-4:** As a developer, I want to see only essential messages in Slack (command started, input needed, command finished, and document drafts for review) so that my Slack is not flooded with verbose agent output.

**US-5:** As a developer, I want to run `/feature` for a designed feature from Slack and have it execute the full pipeline (dev, QA, fix, commit) in the existing worktree so that I can kick off long builds from mobile and check results later.

**US-6:** As a developer, I want to check what sessions are currently active by typing `@bot status` so that I know what is running.

**US-7:** As a developer, when a spec finishes, I want the bridge to ask me "Run /design now?" in the thread so that I can continue to the next stage without remembering the exact command or worktree.

**US-8:** As a developer, when I say "yes" to continuing to the next stage, I want a new Slack thread to be created for that stage so that each stage's conversation is cleanly separated.

**US-9:** As a developer, I want the branch to be automatically pushed to the remote when the final stage (feature pipeline) completes so that I can go straight to creating a PR or reviewing on GitHub.

**US-10:** As a developer, I want all stages of a feature (spec, design, feature) to operate on the same worktree and branch so that the work is unified from idea through implementation.

**US-11:** As a developer, I want commands to run in the foreground (not spawn subagents) so that the bridge can stream output directly and I receive timely updates in Slack.

**US-12:** As a developer, I want `/design` to create a worktree if one does not already exist so that I can start a feature from the design stage if I prefer to skip a formal spec.

**US-13:** As a developer on my phone, I want the full spec markdown to appear in the Slack thread when the PM agent finishes a draft so that I can read and review it without opening a file or accessing my computer.

**US-14:** As a developer on my phone, I want the full design markdown to appear in the Slack thread when the Architect agent finishes a draft so that I can review and approve or give feedback directly in Slack.

## Functional Requirements

### Command Handling

- FR-1: The bridge must accept commands via Slack mentions in the format `@bot /command "arguments"` where `/command` is any command from the existing workflow toolkit: `/spec`, `/design`, `/feature`, `/fix`, `/dev-task`, `/qa-review`, `/dev-fix`, `/worktrees`, `/pr`.
- FR-2: Each command invocation must create a new Slack thread (or use the thread where it was invoked). All subsequent messages for that session must stay within that thread.
- FR-3: Each Slack thread must map to exactly one isolated session. There must be no shared state or context between sessions in different threads.
- FR-4: When a user replies in a thread that has an active session, that reply must be forwarded as input to the running session.

### Worktree Lifecycle

- FR-5: When `/spec "feature name"` is invoked, the bridge must create a new worktree and branch for that feature within the parent project directory before the spec command begins execution. The spec must be written inside that worktree.
- FR-6: When `/design` is invoked for a feature that already has a worktree (created during spec), the design command must execute inside that existing worktree.
- FR-7: When `/feature` is invoked for a feature that already has a worktree, the feature pipeline must execute inside that existing worktree.
- FR-8: The bridge must track which worktree/branch belongs to which feature so that subsequent stages can find and reuse it.
- FR-9: When the final stage of a feature pipeline completes successfully, the bridge must push the feature branch to the remote.
- FR-25: When `/design "feature name"` is invoked and no worktree yet exists for that feature, the bridge must create a new worktree and branch for that feature before the design command begins execution. This allows users to start from the design stage without requiring a prior spec.

### Stage Transitions

- FR-10: When a command completes and there is a logical next stage (e.g., spec -> design, design -> feature), the bridge must ask the user in the current thread whether they want to proceed to the next stage.
- FR-11: If the user confirms they want to proceed, the bridge must create a new Slack thread for the next stage and run the next command there, using the same worktree.
- FR-12: The new thread's initial message must reference the previous stage (e.g., "Starting /design for 'user authentication' -- continuing from spec in [link to previous thread]").

### Session Management

- FR-13: A `status` command must list all currently active sessions with their thread, command, associated worktree/branch, and how long they have been running.
- FR-14: Sessions must automatically expire after 30 minutes of inactivity (no user input and no output).
- FR-15: Multiple sessions must be able to run concurrently across different threads.
- FR-16: The bridge must connect to Slack using Socket Mode (no public URL or ingress required).
- FR-17: The bridge must run inside the project's dev container, operating on the parent project directory.

### Message Behavior

- FR-18: The bridge must post a Slack message at exactly these points during command execution: (a) when a command starts (including the worktree/branch being used), (b) when user input is required, (c) when a command completes (with a summary of the outcome), and (d) when a spec or design document is ready for review (see FR-26).
- FR-19: No other messages should be posted -- no verbose agent logging in Slack.
- FR-20: The bridge must respect Slack API rate limits and not flood the workspace with messages.
- FR-21: When a session requires user input, the Slack message must clearly indicate what input is expected (e.g., "The PM agent is asking: [question]").

### Command and Agent Modifications

- FR-22: All commands that currently spawn subagents must be modified to run in the foreground (main thread) so that the CLI process can stream output directly.
- FR-23: Commands must each handle exactly one task per invocation. Bulk/parallel modes (e.g., passing multiple task IDs or a count to `/spec`) must be removed.
- FR-24: The `/spec` command must create a worktree and branch as part of its execution (consistent with FR-5).
- FR-25: (See Worktree Lifecycle above -- `/design` must also create a worktree if none exists.)

### In-Slack Document Review

- FR-26: When the PM agent finishes writing or revising a spec draft, the bridge must post the full markdown content of the spec into the Slack thread.
- FR-27: When the Architect agent finishes writing or revising a design draft, the bridge must post the full markdown content of the design into the Slack thread.
- FR-28: If a document exceeds Slack's message size limit, the bridge must split it across multiple sequential messages, preserving reading order, so the full document is visible in the thread.
- FR-29: The document post must be clearly labeled (e.g., "Spec Draft v1 for 'user authentication'" or "Design Draft v2 for 'user authentication'") so the user knows what they are reviewing.

## Edge Cases

- User sends a command while another command is already running in the same thread -- the bridge must reject the second command with a clear message ("A session is already active in this thread. Please wait for it to finish or start a new thread.").
- User replies in a thread after the session has expired -- the bridge must inform the user that the session has timed out and suggest starting a new command.
- User sends an unrecognized command -- the bridge must respond with a list of supported commands.
- The underlying process crashes or exits unexpectedly mid-session -- the bridge must detect this, notify the user in the thread that the session ended unexpectedly, and clean up the session. The worktree must remain intact (not deleted) so work is not lost.
- Slack message exceeds Slack's character limit (4000 chars for regular messages, approximately 3000 per block) -- output must be split across multiple messages preserving reading order.
- User sends a command with no arguments where arguments are required -- the bridge must respond with usage guidance for that command.
- User runs `/feature` but no worktree exists for that feature yet -- the bridge must inform the user that the feature needs a spec or design first, or allow them to specify which worktree to use.
- User runs `/spec` for a feature name that already has an existing worktree -- the bridge must detect this and ask the user whether to reuse the existing worktree or create a new one.
- The branch push fails after feature completion (e.g., remote is unreachable) -- the bridge must notify the user that the push failed, indicate the work is still in the local worktree, and suggest retrying.
- User declines to continue to the next stage when prompted -- the bridge must acknowledge the choice and leave the worktree in place for later use. The user can invoke the next stage manually at any time.
- User runs `/design` for a feature name that already has an existing worktree -- the bridge must reuse that worktree (same behavior as v2 FR-6). If no worktree exists, it creates one per FR-25.
- A spec or design document is very large and must be split across many Slack messages -- the bridge must number or label the parts (e.g., "Part 1 of 3", "Part 2 of 3") so the user can follow the reading order.
- The bridge successfully posts the first part of a split document but fails to post a subsequent part (e.g., rate limit) -- the bridge must inform the user that the full document was not delivered and indicate that the complete file is available in the worktree.
- User attempts to use a bulk/parallel invocation (e.g., `/spec 3` or `/spec PROJ-123,PROJ-456`) -- the bridge must reject with a message explaining that each command handles one task at a time, and suggest running separate commands.

## UX Constraints

- The bridge must not require any configuration in Slack beyond installing the bot and adding it to channels.
- Command syntax in Slack must mirror the existing CLI syntax as closely as possible (e.g., `/spec "user auth"` in CLI becomes `@bot /spec "user auth"` in Slack).
- Thread replies to the bot must behave identically to typing input in the CLI -- no special syntax required.
- The "input needed" message must include the actual question or prompt from the agent, not just a generic "input needed" notification.
- The completion message must include a meaningful summary (e.g., "Spec written to docs/specs/user-auth/v1.md on branch feature/user-auth" or "Feature pipeline complete: 12 files changed, QA passed, committed and pushed on feature/user-auth").
- The stage transition prompt must be a simple, clear question (e.g., "Spec complete. Run /design for 'user authentication' now?") that the user can answer conversationally (yes, no, y, n, etc.).
- When a new thread is created for the next stage, the user must be notified in the old thread with a link to the new thread so they can follow along.
- The start message for each command must include which worktree/branch is being used, so the user always knows where work is happening.
- The bridge must require only Slack credentials and the parent project directory path to operate -- no other configuration for MVP.
- Manual restart is acceptable after container restarts. No auto-start requirement for MVP.
- Spec and design documents posted in Slack must be formatted as readable markdown text, not as file attachments or code blocks, so they render naturally in Slack on all devices including mobile.
- When a document is split across multiple messages, each part must be posted in order with clear labels so the user can read sequentially.
- The document review message must be visually distinct from other bridge messages (e.g., labeled with the document type and version) so the user immediately recognizes it as content to review.

## Acceptance Criteria

- [ ] AC-1: User types `@bot /spec "feature name"` in a Slack channel. A worktree and branch are created for that feature. The spec workflow runs inside that worktree. The user receives start, input-needed, and completion messages in a thread.
- [ ] AC-2: User replies in the thread to answer agent questions, and the conversation continues as it would in the CLI.
- [ ] AC-3: When a spec completes, the bridge asks in the thread "Run /design now?" The user replies yes. A new Slack thread is created for the design stage, running in the same worktree.
- [ ] AC-4: When a design completes, the bridge asks "Run /feature now?" The user replies yes. A new thread is created for the feature pipeline, running in the same worktree.
- [ ] AC-5: When the feature pipeline completes successfully, the branch is automatically pushed to the remote. The completion message includes the branch name.
- [ ] AC-6: Running two commands for two different features in two different threads simultaneously both complete successfully without interfering with each other, each in their own worktree.
- [ ] AC-7: A session that receives no interaction for 30 minutes is automatically cleaned up, and a subsequent reply in that thread informs the user.
- [ ] AC-8: `@bot status` returns a list of active sessions including the worktree/branch for each (or "No active sessions").
- [ ] AC-9: The bridge runs inside the dev container and requires only Slack credentials and the parent project directory to operate -- no public URLs, no cloud deployment.
- [ ] AC-10: Slack messages from the bridge are limited to start, input-needed, completion, and document-review events -- no verbose agent logging appears in Slack.
- [ ] AC-11: If the underlying process crashes, the user is notified in the thread that the session ended unexpectedly. The worktree is preserved.
- [ ] AC-12: User declines to continue to the next stage. The worktree remains intact. The user can later invoke the next stage manually for that feature.
- [ ] AC-13: User runs `/feature` for a feature that has no worktree yet. The bridge responds with a clear message that the feature needs a spec or design first (or asks the user to specify a worktree).
- [ ] AC-14: Commands run in the foreground (main thread) without spawning subagents. The bridge receives streamed output directly from the CLI process.
- [ ] AC-15: User attempts a bulk invocation (e.g., `@bot /spec 3`). The bridge rejects it with a message explaining one task per command.
- [ ] AC-16: User types `@bot /design "feature name"` when no worktree exists for that feature. A worktree and branch are created, and the design workflow runs inside it.
- [ ] AC-17: When the PM agent finishes a spec draft, the full spec markdown appears in the Slack thread. The user reads it and replies with feedback. The PM agent incorporates the feedback and posts the revised spec in the thread.
- [ ] AC-18: When the Architect agent finishes a design draft, the full design markdown appears in the Slack thread. The user replies "approved" and the workflow proceeds.
- [ ] AC-19: A spec document that exceeds Slack's message size limit is split across multiple labeled messages (e.g., "Part 1 of 3") that appear in reading order in the thread.
- [ ] AC-20: The bridge fails to deliver a subsequent part of a split document. The user is informed that the full document was not posted and is told where to find the file in the worktree.
