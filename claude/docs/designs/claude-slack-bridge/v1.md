# Claude Code Slack Bridge — Architecture Design

**Design Version:** v1
**PM Spec:** docs/specs/claude-slack-bridge/v3.md (Spec v3)
**Status:** Draft
**Last Updated:** 2026-02-26

---

## Scope Confirmation

### What is being implemented

A Node.js/TypeScript bridge that connects Slack to the Claude Code CLI, allowing users to trigger workflow commands (`/spec`, `/design`, `/feature`, etc.) from Slack and interact with running sessions via thread replies. The bridge spawns `claude` CLI processes using `-p` mode with `--output-format stream-json`, parses the output stream for structured markers, and posts filtered messages (start, input-needed, completion, document-review) to Slack threads. Existing command and agent markdown files are modified to run foreground (no subagents), remove bulk mode, create worktrees earlier, and emit structured markers the bridge can detect.

### Explicitly out of scope

- Multi-user or multi-workspace support
- Persistent session state across bridge restarts
- Auto-start or orchestrated startup with dev container
- Per-channel project directory mapping
- New commands beyond the existing workflow toolkit
- Web UI or any interface besides Slack
- Access control or user permissions

## Proposed System Changes

### New: Bridge Application (`bridge/`)

A new Node.js/TypeScript application inside this repository.

**File structure:**

```
bridge/
  package.json
  tsconfig.json
  src/
    index.ts              # Entry point: init Bolt app, register handlers
    slack.ts              # Slack Bolt app setup, Socket Mode connection
    session-manager.ts    # Session lifecycle: create, track, expire, lookup
    cli-runner.ts         # Spawn claude CLI, parse stream-json, manage resume
    stream-parser.ts      # NDJSON parser: extract markers, text deltas, result
    message-poster.ts     # Format and post messages to Slack (splitting, labels)
    worktree-tracker.ts   # In-memory feature-name → worktree/branch mapping
    command-router.ts     # Parse @bot messages, validate, route to handler
    config.ts             # Env var loading and defaults
  .env.example            # Template for required env vars
```

### Modified: Command Markdown Files

Each command file that currently uses subagents is modified to:

1. **Remove bulk/parallel mode** — delete Steps 2 and 4 (multi-task flow) from `/spec` and `/design`; delete multi-feature parallel flow from `/feature`. Each command handles exactly one task.
2. **Inline agent instructions** — replace `Use the [agent] agent` / Task tool invocations with direct inline instructions. The agent's role, rules, and behavior are copied into the command flow so everything executes in the main thread.
3. **Add structured markers** — emit specific text patterns at key moments that the bridge parses from the stream.
4. **Remove interactive selection prompts** — replace any arrow-key / numbered-choice patterns with plain text questions that accept text replies.
5. **Add worktree creation** — `/spec` already creates worktrees; `/design` gains worktree creation when none exists.

**Commands requiring modification:**

| Command | Remove bulk | Inline agents | Add markers | Add worktree |
|---------|------------|---------------|-------------|-------------|
| `spec.md` | Yes (Steps 2, 4) | Yes (PM agent) | Yes | Already has it |
| `design.md` | Yes (Steps 2, 4) | Yes (Architect agent) | Yes | Yes (new) |
| `feature.md` | Yes (multi-feature) | Yes (Dev, QA agents) | Yes | No (already has it) |
| `fix.md` | No (already single) | Yes (Dev, QA agents) | Yes | No |
| `dev-task.md` | No (already single) | Yes (Dev agent) | Yes | No |
| `qa-review.md` | No | Yes (QA agent) | Yes | No |
| `dev-fix.md` | No | Yes (Dev agent) | Yes | No |

**Agent files (`agents/*.md`) are NOT modified.** They remain as reference definitions. Their instructions are inlined into the command files.

### Bridge-CLI Protocol: Structured Markers

The modified commands emit these text markers at specific points. The bridge scans the streamed text output for these patterns.

```
[BRIDGE:START] command=/spec feature="user authentication" worktree=.worktrees/user-auth branch=feature/user-auth
[BRIDGE:INPUT_NEEDED] question="What authentication providers should be supported?"
[BRIDGE:DOCUMENT_READY] type=spec version=v1 path=docs/specs/user-auth/v1.md
[BRIDGE:DOCUMENT_READY] type=design version=v1 path=docs/designs/user-auth/v1.md
[BRIDGE:STAGE_COMPLETE] command=/spec next=/design feature="user authentication"
[BRIDGE:ERROR] message="Worktree creation failed: branch already exists"
```

Rules for markers:
- Each marker is on its own line, always prefixed with `[BRIDGE:` and suffixed with `]`
- Key-value pairs are space-separated, values with spaces are double-quoted
- The bridge strips markers from any text posted to Slack
- Commands that run in the CLI without the bridge simply ignore markers (they appear as harmless text output)

When a command has questions for the user (e.g., PM clarifying questions, architect questions), it emits `[BRIDGE:INPUT_NEEDED]` with the question text. The command then ends its turn (the CLI process outputs a `result` message and exits). The bridge posts the question to Slack and waits for a thread reply, then resumes the session with `--resume`.

When a command writes a spec or design file, it emits `[BRIDGE:DOCUMENT_READY]` with the file path. The bridge reads the file from disk and posts its content to the Slack thread.

## Data Flow

### Session Lifecycle

1. **User posts** `@bot /spec "user authentication"` in a Slack channel
2. **Command router** parses the mention, extracts command (`/spec`) and args (`"user authentication"`)
3. **Session manager** creates a new session, records thread_ts and channel
4. **CLI runner** spawns: `claude -p "Run the /spec workflow for 'user authentication'. [full inlined command instructions or reference to command file]" --output-format stream-json --dangerously-skip-permissions [--allowedTools ...]`
5. **Stream parser** reads stdout line by line (NDJSON), looking for:
   - `{type: "system", subtype: "init"}` — capture `session_id` for `--resume`
   - `{type: "stream_event", event: {type: "content_block_delta"}}` — accumulate text, scan for `[BRIDGE:*]` markers
   - `{type: "result"}` — CLI turn complete
6. **On `[BRIDGE:START]`** — post start message to Slack thread with worktree/branch info
7. **On `[BRIDGE:INPUT_NEEDED]`** — post the question to Slack thread; mark session as `awaiting_input`
8. **On user reply in thread** — session manager finds the session by thread_ts; CLI runner spawns: `claude -p "<user's reply>" --resume <session_id> --output-format stream-json --dangerously-skip-permissions`
9. **On `[BRIDGE:DOCUMENT_READY]`** — read file from disk, post content to Slack (splitting if needed)
10. **On `[BRIDGE:STAGE_COMPLETE]`** — post completion summary + stage transition prompt ("Run /design now?")
11. **On user reply "yes" to stage transition** — create new thread in same channel, start new session with next command, link old thread to new thread

### CLI Interaction Model

The bridge does NOT maintain a long-running interactive CLI process. Instead:

- Each "turn" is a separate `claude -p` invocation
- The first turn uses `-p` with the full prompt
- Subsequent turns use `-p` with the user's reply text plus `--resume <session_id>`
- The CLI process starts, streams output, and exits when its turn is done
- The bridge captures the `session_id` from the `system/init` message for resumption

This is important: there is no stdin interaction. Each CLI invocation is a complete request-response cycle.

### Working Directory

The bridge sets `cwd` on the spawned process to:
- The parent project directory for initial commands
- The worktree directory once a worktree is created (the command itself creates the worktree and emits the path in `[BRIDGE:START]`)

Since worktree creation is handled by the commands, the bridge starts all processes in the parent project directory. The command creates the worktree, emits `[BRIDGE:START]` with the worktree path, and subsequent `--resume` calls for that session use the worktree path as cwd.

## Session Management

### Session Object (in-memory)

```
Session {
  id: string                    // unique session ID (uuid)
  threadTs: string              // Slack thread timestamp
  channelId: string             // Slack channel ID
  command: string               // e.g., "/spec"
  featureName: string           // e.g., "user authentication"
  cliSessionId: string | null   // captured from stream init message
  worktreePath: string | null   // captured from BRIDGE:START marker
  branchName: string | null     // captured from BRIDGE:START marker
  status: "running" | "awaiting_input" | "completed" | "error" | "expired"
  startedAt: Date
  lastActivityAt: Date          // updated on any output or input
  activeProcess: ChildProcess | null  // reference to current CLI process
}
```

### Session Map

```
Map<threadTs, Session>          // primary lookup: thread → session
Map<featureName, WorktreeInfo>  // worktree tracker: feature → {worktreePath, branchName}
```

The worktree tracker persists feature-to-worktree mappings across sessions so that stage transitions (spec -> design -> feature) can find the existing worktree even though each stage is a new session/thread.

### Expiry

A timer runs every 60 seconds. Sessions with `lastActivityAt` older than 30 minutes and status `awaiting_input` are marked `expired`. No cleanup of worktrees -- they remain on disk.

### Concurrency

Max concurrent sessions is controlled by `MAX_CONCURRENT_SESSIONS` (default: 3). When the limit is reached, new command requests receive a message: "Maximum concurrent sessions reached. Please wait for a running session to complete."

## Message Posting to Slack

### Filtering

The bridge posts messages to Slack ONLY when it detects these markers:
- `[BRIDGE:START]` — "Starting /spec for 'user authentication' on branch feature/user-auth (worktree: .worktrees/user-auth)"
- `[BRIDGE:INPUT_NEEDED]` — "The PM agent is asking: [question text]"
- `[BRIDGE:DOCUMENT_READY]` — posts the document content (see below)
- `[BRIDGE:STAGE_COMPLETE]` — "Spec complete. Run /design for 'user authentication' now?"
- `[BRIDGE:ERROR]` — "Error: [message]"
- Process exit with non-zero code — "Session ended unexpectedly. Your worktree is preserved at .worktrees/user-auth."

All other streamed text (tool calls, reasoning, file reads/writes) is silently discarded.

### Document Posting (FR-26, FR-27, FR-28)

When `[BRIDGE:DOCUMENT_READY]` is detected:

1. Read the file at the specified path from disk
2. Prepend a label: "**Spec Draft v1 for 'user authentication'**" (or Design Draft)
3. If total length <= 3000 characters: post as a single message
4. If > 3000 characters: split on the nearest markdown heading (`## `) boundary that keeps each part under 3000 chars. If no heading boundary fits, split at 3000 chars on a newline boundary
5. Label each part: "Part 1 of N", "Part 2 of N", etc.
6. Post parts sequentially with a small delay (200ms) to maintain order and respect rate limits
7. If any part fails to post: inform user that the full document was not delivered and provide the file path in the worktree

### Status Command

`@bot status` is handled directly by the bridge (no CLI process). It iterates the session map and returns a formatted list of active sessions.

## Stage Transition Flow

1. Command completes, emits `[BRIDGE:STAGE_COMPLETE] command=/spec next=/design feature="user authentication"`
2. Bridge posts in current thread: "Spec complete. Run /design for 'user authentication' now?"
3. User replies "yes" (matched by: yes, y, yeah, sure, do it, go ahead, etc.)
4. Bridge posts in current thread: "Starting /design in a new thread..."  with a link to the new thread
5. Bridge posts a new top-level message in the same channel: "Starting /design for 'user authentication' -- continuing from spec in [link to previous thread]"
6. New session created, mapped to new thread_ts
7. CLI spawned with `/design` command prompt, cwd set to the existing worktree path (looked up from worktree tracker by feature name)
8. If user replies "no" -- bridge acknowledges, session ends, worktree stays

Stage chain: `/spec` -> `/design` -> `/feature`. The bridge only offers the next stage, never skips.

## Command Modifications Detail

### What changes in `/spec` (spec.md)

**Remove:** Steps 2, 4 (multi-task flow), Step 3's subagent spawning, numbered selection prompts
**Add:** Inline PM agent instructions directly. Add markers:
- Emit `[BRIDGE:START]` after worktree creation
- Emit `[BRIDGE:INPUT_NEEDED]` before each question round
- Emit `[BRIDGE:DOCUMENT_READY]` after writing the spec file
- Emit `[BRIDGE:STAGE_COMPLETE]` after spec is approved
**Keep:** Single-task flow (Step 3), worktree creation, Linear integration, review loop

### What changes in `/design` (design.md)

**Remove:** Steps 2, 4 (multi-task flow), Step 3's subagent spawning
**Add:** Inline Architect agent instructions. Add markers (same set as /spec). Add worktree creation: if no worktree exists for the feature, create one before running (same logic as /spec).
**Keep:** Single-task flow (Step 3), Q&A phases, review loop

### What changes in `/feature` (feature.md)

**Remove:** Multi-feature parallel flow, base branch prompt (bridge always uses main or the existing worktree's branch)
**Add:** Inline Dev/QA/DevFix agent instructions. Add markers:
- `[BRIDGE:START]` at pipeline start
- `[BRIDGE:STAGE_COMPLETE]` at pipeline end (no "next" stage -- feature is terminal)
- `[BRIDGE:ERROR]` on escalation or max fix cycles
**Keep:** Single-feature flow, stages 1-4, existing worktree detection, feedback cycle

### What changes in `/fix`, `/dev-task`, `/qa-review`, `/dev-fix`

**Remove:** Subagent spawning
**Add:** Inline the relevant agent instructions. Add `[BRIDGE:START]` and `[BRIDGE:STAGE_COMPLETE]` markers. These commands do not have multi-task modes, so changes are smaller.

## Prompt Construction

When the bridge spawns a CLI process, it needs to pass the full command prompt. Two options:

**Option A (chosen): Reference the command file.** The bridge passes a prompt like:
```
Follow the instructions in /path/to/commands/spec.md to write a PM spec for "user authentication".
```
The CLI reads the file and executes. This keeps the bridge thin and the command files as the source of truth.

**Option B (rejected): Embed command content in the prompt.** The bridge reads the command markdown and pastes it into the `-p` argument. This duplicates content and makes prompts enormous.

Option A is simpler. The command files already contain all instructions. The bridge just tells the CLI which command file to follow and what arguments to use.

## Permission and Security Model

- The bridge runs inside a dev container with no public network exposure
- Socket Mode means no inbound HTTP -- Slack connects outbound only
- `--dangerously-skip-permissions` is used on every CLI spawn (acceptable in dev container)
- `--allowedTools` is configurable via `ALLOWED_TOOLS` env var for defense-in-depth (e.g., `Read,Write,Edit,Glob,Bash`)
- Single-user, single-workspace -- no auth beyond the Slack bot token
- The bot should only respond to mentions in the configured channel (or all channels it is added to)
- No secrets are logged or posted to Slack

## Edge Case Handling Strategy

- **Command while session active in thread** — session manager checks thread_ts before creating session; rejects with message
- **Reply after session expired** — session manager returns null for expired thread; bridge informs user
- **Unrecognized command** — command router returns list of supported commands
- **CLI process crashes** — `child_process` 'exit' event with non-zero code triggers error message in thread; session marked `error`; worktree preserved
- **Message exceeds Slack limit** — document splitter handles this (see Message Posting section)
- **No arguments provided** — command router checks for required args per command; returns usage guidance
- **`/feature` with no worktree** — the inlined command logic handles this (checks for worktree, reports if missing)
- **`/spec` with existing worktree** — the inlined command logic handles this (asks user whether to reuse)
- **Push failure** — CLI runner catches git push errors from stream; bridge posts failure notification with worktree path
- **User declines stage transition** — bridge acknowledges, marks session completed
- **Split document partial failure** — bridge tracks parts posted; on failure, informs user and provides file path
- **Bulk invocation attempt** — command router detects patterns like `/spec 3` or comma-separated IDs; rejects with explanation

## Tradeoffs Considered

**Why `-p` per turn instead of interactive stdin:**
The Claude CLI's `-p` flag is non-interactive and does not support stdin. Using `--resume` for follow-ups is the only supported pattern. This means each user reply spawns a new CLI process, which adds ~1-2s of startup overhead per turn. The alternative (interactive mode with stdin/stdout) would require slash commands to work, but they do not work with `-p`. The overhead is acceptable for a mobile-first, async workflow.

**Why markers in text output instead of a separate control channel:**
The `stream-json` format has no extensible event type for custom signals. Embedding markers in the assistant's text output is the simplest approach that requires no CLI modifications. The downside is that markers are visible if the user runs the modified commands directly in the terminal -- but they are clearly bracketed and ignorable.

**Why inline agent instructions instead of keeping subagents:**
Subagents cannot relay input back to the parent. Since the bridge uses `-p` (non-interactive), all input must flow to the single running process. Inlining removes the subagent boundary so questions reach the bridge directly. The cost is larger command files, but the commands remain self-contained markdown.

**Why Option A (reference command file) for prompts:**
Keeps the bridge code minimal and the command files as the single source of truth. If command files change, the bridge picks up changes automatically. The risk is that Claude may not follow the file instructions as faithfully as embedded instructions, but this is mitigated by explicit "Follow the instructions in [file]" framing.

**Why in-memory session state:**
The spec explicitly requires this for MVP. Adding persistence (SQLite, JSON file) would be premature. Worktrees on disk serve as durable state; sessions are ephemeral coordination.

## Risks

- **Marker reliability:** Claude may not emit markers consistently or may alter their format. Mitigation: command files include explicit instructions to emit exact marker strings; integration tests verify marker output.
- **`--resume` behavior:** The `--resume` flag reconnects to a prior session's context. If the session has expired server-side (Anthropic's session TTL), resume will fail. Mitigation: catch resume errors and inform the user to restart the command.
- **Command file size:** Inlining agent instructions into commands will roughly double their size. This increases token consumption per invocation. Mitigation: inline only the essential rules, not the full agent file verbatim.
- **Prompt reference fidelity:** Telling Claude "follow the instructions in this file" relies on Claude reading and following the file completely. If Claude summarizes or skips sections, behavior will diverge. Mitigation: integration testing; if unreliable, fall back to Option B (embed content).
- **Rate limits:** Both Slack API rate limits (1 msg/sec per channel) and Anthropic API rate limits. Mitigation: 200ms delay between Slack messages; max concurrent sessions cap.
- **Worktree tracker drift:** The in-memory worktree tracker is lost on restart. A user who ran `/spec` before a restart will need to manually specify the worktree for `/design`. Mitigation: the command files scan `.worktrees/` to discover existing worktrees.

## Complexity Control

**Abstractions intentionally NOT introduced:**
- No message queue or event bus -- direct function calls between components
- No database or ORM -- plain Maps for session and worktree tracking
- No plugin system for commands -- command routing is a simple switch/map
- No middleware chain for Slack events -- Bolt's built-in event handling is sufficient
- No abstraction over the CLI invocation -- it is always `claude -p ... --resume ... --output-format stream-json`

**What is explicitly deferred:**
- Persistent session state (post-MVP)
- Multi-user support (post-MVP)
- Auto-start with dev container (post-MVP)
- Per-channel project mapping (post-MVP)
- Graceful handling of Anthropic session expiry (detect and report, but no auto-retry of full workflow)

## Testing Considerations

**Unit:**
- `stream-parser.ts`: given sample NDJSON lines, correctly extracts markers and ignores non-marker text
- `command-router.ts`: parses various command formats, rejects bulk patterns, returns correct command/args
- `message-poster.ts`: splits documents correctly at heading boundaries, labels parts
- `session-manager.ts`: creates sessions, enforces concurrency limit, expires stale sessions

**Integration:**
- Spawn a real `claude` CLI process with a test prompt, verify stream-json output is parsed correctly
- Verify marker detection end-to-end: modified command emits marker, bridge detects it, posts correct Slack message
- Verify `--resume` flow: first invocation captures session_id, second invocation resumes successfully
- Verify document posting: write a test markdown file, trigger document-ready, verify Slack receives correct content (split if needed)

**Security:**
- Verify `--dangerously-skip-permissions` is always passed
- Verify `--allowedTools` is passed when configured
- Verify no secrets appear in Slack messages
- Verify the bridge only responds to mentions (not all messages in channel)

## Migration Plan

No data or schema migration required. This is a new application.

**Rollout steps:**
1. Modify command markdown files (add markers, inline agents, remove bulk)
2. Build and test bridge application
3. Create Slack app, obtain bot token and app token
4. Configure env vars in dev container
5. Start bridge: `cd bridge && npm start`
6. Test in Slack channel

**Rollback:** Stop the bridge process. Modified command files still work in the CLI (markers are harmless text). To fully revert, restore command files from git.

**Backward compatibility of command changes:** The modified commands work in both contexts:
- In the CLI: markers appear as text output but do not affect functionality; single-task-only is a behavior change users must adapt to (no more `/spec 3`)
- Via the bridge: markers are parsed and stripped; single-task is the expected mode
